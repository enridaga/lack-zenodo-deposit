# Case Studies — Session Notes

## Overview

This session established a process for selecting, extracting, and publishing use case paths from the LACK Knowledge Graph as a Case Studies section on the LACK website.

## What a Case Study Is

Each case study is a two-hop path of the form:

**Entity → [relation] → Proxy → [relation] → Target**

These paths connect entities that were previously disconnected in the KG, discovered by the graph's inference engine. They demonstrate the value of the KG by revealing non-obvious connections between actors in the climate lobbying network.

## Selection Process

1. Read `paths-annotated.csv` — a file containing resolved (human-readable) two-hop paths between previously disconnected entities, with labels for all nodes and relations.
2. Sample the file in chunks (head, tail, middle) to get a varied view of available paths.
3. Filter out paths where the second hop is a `sameAs → Wikidata/DBpedia` link, as these are entity alignment paths rather than substantive connections.
4. Select paths that have a genuinely named target entity and tell an interesting story about actor relationships in the climate lobbying network.
5. Aim for diversity of angle across the selected set (e.g. funding chains, industry–industry links, lobbying-to-policy, legal–judicial, governance–civil society, person–institution–person).

## Output: `content/case-studies.csv`

The selected paths are recorded as rows in `content/case-studies.csv` (also copied to the LobbyGraph Google Drive input folder), with columns:

`entity, entity_label, relation, relation_label, proxy, proxy_label, relation1, relation1_label, target, target_label`

## Technical Pipeline

### Components

**`content/case-studies.csv`**
The source of truth for which case studies are active. Each row is one case study path. Edit this file to add, remove, or change case studies.

**`generate-case-studies.py`**
A standalone script that reads `content/case-studies.csv` and queries `KG.ttl` to produce `case-studies.ttl` — a small, self-contained TTL file containing all entity metadata, base triples, and reified statement provenance needed for the website pages. Run this script whenever `content/case-studies.csv` changes.

**`case-studies.ttl`**
The intermediate data file consumed by the build step. Contains labels, types, sameAs links, and all `rdf:Statement` instances (with `prov:wasDerivedFrom`, `dct:source`, `lack:since`, `lack:until`) for the two hops of each case study.

**`content/case-study/cs-N.md`**
One stub file per case study, containing only frontmatter with the `case_study_id`. The actual page content is generated by the build step from `case-studies.ttl`.

**`content/case-studies.md`**
The index page listing all case studies. Appears in the site nav under "Case Studies".

**`build.py`**
The standard site builder. When run, it reads `case-studies.ttl` and generates `_site/case-study/cs-N.html` for each stub file, as well as `_site/case-studies.html` for the index. Each case study page contains a path visualisation with entity links to the Explore page, and two evidence tables (one per hop) with source links.

### Execution Order

```bash
# Step 1 — run once, or whenever case-studies.csv changes
python3 generate-case-studies.py

# Step 2 — run to rebuild the full site
python3 build.py
```

### Entity Links

Each entity in the path and evidence tables links to the LACK Explore page using the fragment URL format:

```
https://climatesense-project.eu/lack/static/explore.html#entity=<percent-encoded URI>
```
